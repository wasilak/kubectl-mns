<objective>
Add per-namespace output labels (OUTPUT-01), per-namespace error continuation (ERRORS-01), and forwarding of `--context` (ARGS-01) and `--kubeconfig` (ARGS-02) to every kubectl call.

Purpose: Make multi-namespace output legible, prevent a single namespace failure from aborting the run, and support multi-cluster / non-default kubeconfig usage.
Output: Updated `kubectl-mns` script implementing all four Phase 2 requirements; passes shellcheck cleanly.

This is a REVISED plan incorporating cross-AI review feedback (02-REVIEWS.md). Key revisions vs. the original 02-01-PLAN.md:

- **Label placement fix (HIGH, OpenCode):** The namespace label is now printed AFTER successful data capture, not before the kubectl call. This prevents a "dangling label with no data" on a failing namespace.
- **Stderr conflation fix (HIGH, both reviewers):** The kubectl call no longer uses `2>&1`. On success, kubectl's stderr (warnings, deprecation notices) passes through live to the user's stderr instead of being captured into `$data` and printed on stdout. On failure, stderr also flows live; the error message just names the failing namespace.
- **Post-loop `consume_next_as` guard (MEDIUM, OpenCode):** Detects `kubectl-mns --context` (flag without value) and exits with a clear error instead of falling through to the generic usage message.
- **All-fail exit code documented (MEDIUM, OpenCode):** Behavior contract now explicitly states the script exits 0 even when all namespaces fail.
- **New smoke tests F and G (both reviewers):** Empty-data-on-success and all-namespaces-fail scenarios.
</objective>

The post-Phase-2 `run_kubectl()` after revision (supersedes 02-PATTERNS.md lines 286–344 where they conflict):

  function run_kubectl() {
      is_double_dash="false"
      consume_next_as=""
      namespaces=()
      actual_kubectl_args=()
      extra_kubectl_flags=()

      for item in "$@"; do

        if [[ -n "$consume_next_as" ]]; then
          extra_kubectl_flags+=("$consume_next_as" "$item")
          consume_next_as=""
          continue
        fi

        if [ "--" == "$item" ]; then
          is_double_dash="true"
          continue
        fi

        if [ "false" == "$is_double_dash" ]; then
          case "$item" in
            --context|--kubeconfig)
              consume_next_as="$item"
              ;;
            *)
              namespaces+=("$item")
              ;;
          esac
        else
          if [[ "$item" != "--all-namespaces" && "$item" != "-A" ]]; then
            actual_kubectl_args+=("$item")
          fi
        fi

      done

      if [[ -n "$consume_next_as" ]]; then
        printf 'Error: %s requires a value\n' "$consume_next_as" >&2
        exit 1
      fi

      if [ ${#namespaces[@]} -eq 0 ]; then
        namespaces+=("default")
      fi

      if [ ${#actual_kubectl_args[@]} -eq 0 ]; then
        usage && exit 1
      fi

      for ns in "${namespaces[@]}"; do
        local kubectl_cmd=("kubectl" "${extra_kubectl_flags[@]}" "${actual_kubectl_args[@]}" --namespace "$ns")
        if ! data=$("${kubectl_cmd[@]}"); then
          printf 'Error: kubectl failed for namespace %s\n' "$ns" >&2
          continue
        fi
        printf '=== namespace: %s ===\n' "$ns"
        if [[ -n "$data" ]]; then
          printf '%s\n\n' "$data"
        fi
      done

  }

Behavior contract (all-fail exit code policy):
- Per-namespace `continue` is unconditional after a kubectl failure. The script does NOT track aggregate failure state.
- If every namespace fails, the script still exits 0. Callers that need a failure exit code on all-fail should treat the presence of `Error: kubectl failed for namespace` lines on stderr as the failure signal.
- Aggregate exit-code policy is explicitly deferred to a future phase.

Stderr semantics (success path):
- On a successful kubectl invocation, kubectl's stderr (warnings, deprecation notices) is NOT captured. It flows live to the script's stderr (i.e., to the user's terminal).
- `$data` contains only stdout. The label and the data printout are stdout-only.

Stderr semantics (failure path):
- On a failing kubectl invocation, kubectl's own stderr ALSO flows live to the script's stderr (same redirection — no special handling).
- The script then prints its OWN one-line summary error `Error: kubectl failed for namespace <ns>` to stderr.
- The summary error does NOT contain captured stderr content; the user has already seen kubectl's real error message live.
- The namespace label is NOT printed on stdout for a failed namespace.
